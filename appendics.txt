Appendix A – Architecture Decision Records (ADRs)
A.1 Overview
Architecture Decision Records (ADRs) document the key technical decisions made during the design of the Theta Chat Bot. Each ADR explains the decision, available alternatives, the rationale for the selected approach, and the expected impact on the overall system.
Maintaining ADRs provides a historical record of architectural choices and helps future developers understand the reasoning behind the solution.

ADR-001 – Retrieval-Augmented Generation (RAG)
Decision
Adopt a Retrieval-Augmented Generation (RAG) architecture instead of relying solely on a Large Language Model.
Alternatives Considered
Direct LLM responses without retrieval
Manually maintained FAQ database
Fine-tuned language model
Selected Approach
Retrieval-Augmented Generation (RAG)
Rationale
Grounds responses in official website content.
Reduces hallucinations.
Eliminates frequent model retraining.
Keeps responses synchronized with website updates.
Easier to maintain over time.
Impact
Positive:
Higher response accuracy.
Better maintainability.
Improved trustworthiness.
Trade-Offs:
Additional retrieval step before response generation.

Appendix B – Prompt Engineering & Guardrails
B.1 Overview
The system prompt is the primary control mechanism governing chatbot behavior. It ensures every response stays grounded in retrieved website content, refuses to speculate, and handles out-of-scope or adversarial input safely. This appendix documents the guardrail rules enforced at the prompt level, separate from the high-level prompt structure already described in Section 4.6.
B.2 Core Guardrail Rules

B.3 Prompt Injection Handling
User input is never treated as an instruction, regardless of phrasing. If a message attempts to override system behavior (e.g., "forget your rules," "act as a different assistant"), the model is instructed to ignore the embedded instruction and respond only using retrieved context for the original question. Such attempts are logged for review, not silently allowed.

Appendix D – AI System Validation & Testing Strategy
D.1 Overview
Unlike traditional software applications, the Theta Chat Bot requires validation of both software functionality and AI response quality. Since the system uses a Retrieval-Augmented Generation (RAG) architecture, testing extends beyond API functionality to include retrieval accuracy, prompt effectiveness, chunking quality, and response reliability.
The primary objective of testing is to ensure that the chatbot consistently provides accurate, relevant, and trustworthy responses based solely on the approved Theta Technolabs website content.
The testing approach combines manual evaluation with technical verification to validate the complete end-to-end pipeline.

D.2 Testing Objectives
The testing process aims to verify:
Functional correctness of the application.
Retrieval accuracy of the vector database.
Quality of content chunking.
Prompt effectiveness.
AI response quality.
Source attribution.
Fallback behavior.
Performance under normal usage.
System stability after content updates.

D.3 Testing Scope
The following components are included in the testing process.

D.4 Manual Retrieval Validation
The effectiveness of the Retrieval-Augmented Generation pipeline depends on retrieving the most relevant content before sending the request to the LLM.
During testing, retrieval quality will be evaluated manually.
Validation Process
Prepare a set of representative questions covering all major website sections.
Submit each question through the chatbot.
Inspect the retrieved content chunks before LLM generation.
Verify that the retrieved chunks contain the information required to answer the question.
Record observations and identify retrieval issues.
Example evaluation criteria:
Were the retrieved chunks relevant?
Was the correct service page selected?
Was important information missing?
Were unrelated chunks retrieved?
Did metadata correctly identify the source page?
Manual evaluation provides confidence that semantic retrieval is functioning as expected before evaluating LLM responses.

D.5 Chunking Validation
Proper chunking directly impacts retrieval quality.
The chunking strategy will be reviewed manually to ensure that content is divided into meaningful semantic units.
Validation criteria include:
Logical paragraph boundaries.
Complete ideas within each chunk.
No abrupt sentence breaks.
Minimal duplication between chunks.
Appropriate chunk size.
Adequate contextual overlap where necessary.
Poor chunking may result in incomplete or inaccurate responses even if retrieval is functioning correctly.

D.6 Prompt Validation
The system prompt plays a significant role in determining chatbot behavior.
Whenever the prompt template, instructions, or response formatting is modified, the complete validation process should be repeated.
Prompt validation should verify:
Adherence to chatbot scope.
Accurate use of retrieved context.
Proper handling of insufficient information.
Consistent response formatting.
Correct citation of website information.
Appropriate fallback behavior.
Prompt modifications should not be deployed without regression testing.

D.7 Response Accuracy Evaluation
After retrieval validation, the generated response should be compared with the corresponding information available on the official website.
Each response should be evaluated using the following criteria.

D.8 Regression Testing
Regression testing ensures that system changes do not negatively impact existing functionality.
Regression testing should be performed after:
Prompt modifications.
Website content updates.
Incremental knowledge synchronization.
Embedding model changes.
LLM provider changes.
Backend code changes.
Chunking strategy updates.
A predefined validation dataset should be used to compare system performance before and after each change.

D.9 Test Dataset

The same dataset should be reused during regression testing to ensure consistent evaluation.

D.10 Performance Testing
The application should also be evaluated from a performance perspective.
Recommended measurements include:
Average response time.
Retrieval latency.
LLM generation time.
End-to-end processing time.
Concurrent request handling.
API throughput.
These measurements help verify that the chatbot meets expected performance objectives.

D.11 Error & Fallback Testing
The chatbot should be tested using unsupported or ambiguous queries to verify fallback behavior.
Example scenarios:
Questions unrelated to Theta Technolabs.
Incomplete user queries.
Invalid input.
Empty messages.
Questions with no matching website content.
Expected behavior:
Do not generate fabricated information.
Inform the user that the requested information is unavailable.
Recommend visiting the relevant website page or contacting Theta Technolabs through the Contact or Inquiry page.

D.12 Acceptance Criteria
The chatbot will be considered ready for deployment when:
Website content is successfully indexed.
Retrieval consistently returns relevant content.
Chunk quality is verified.
AI responses accurately reflect website information.
Prompt behavior is validated.
Fallback responses operate correctly.
Performance meets project expectations.
No critical functional defects remain.

D.13 Continuous Validation
Testing should not conclude after deployment.
Whenever significant changes occur, the validation process should be repeated.
Examples include:
New website content published.
Changes to the chunking strategy.
Prompt updates.
Migration to a different LLM provider.
Major backend changes.
Knowledge synchronization enhancements.
Continuous validation ensures that chatbot quality remains consistent throughout the application's lifecycle.

This scorecard is extremely useful because it measures retrieval quality, chunk quality, prompt behavior, and LLM response quality separately. If something goes wrong, you'll know exactly which stage of the RAG pipeline needs improvement rather than just knowing that "the answer was incorrect."
