  <sheet label="THETA CHATBOT · OVERVIEW" kicker="01 — OVERVIEW" title="Executive Summary" footnum="01">
    <p>Theta ChatBot is a website assistant embedded on <mono>thetatechnolabs.in</mono> that answers visitor questions using <b>only</b> the company's own website content. It does not answer from the open internet and does not let the underlying language model improvise facts about Theta Technolabs — every answer is grounded in a knowledge base built from our own pages, and the model is instructed to say "I don't have that information" rather than guess.</p>
    
    <p>Because the knowledge base is <b>static</b> (it changes only when someone deliberately re-scrapes or edits it — not on every page load), we do not need a heavy real-time crawling pipeline. This lets us keep the system small, cheap to run, and easy for one engineer to operate on our own server.</p>

    <callout label="IN ONE SENTENCE">A visitor's question is converted into a vector, matched against pre-embedded chunks of our own website content stored in Postgres, and the matched chunks are handed to an LLM (Gemini first, OpenAI as a paid fallback) which is told to answer <i>using only that context</i>.</callout>

    <h2>Scope of this document</h2>
    <p>This document is written so that a technical reviewer with no prior context can understand the full system — what is built, why each choice was made over its alternatives, how the pieces talk to each other, and what happens to a single user message from the moment it is typed to the moment a reply appears.</p>

  <sheet label="THETA CHATBOT · METHODOLOGY" kicker="02 — METHODOLOGY" title="Overall Approach" footnum="02">
    <h2>2.1 Why Retrieval-Augmented Generation (RAG)</h2>
    <p>We are <b>not</b> fine-tuning a model on Theta Technolabs data, and we are <b>not</b> letting the LLM answer from its own general knowledge. Both would risk the bot inventing facts about our company (wrong pricing, wrong services, wrong contact details). Instead we use RAG: at answer-time, we retrieve the most relevant pieces of our own website content and paste them into the model's prompt as context, then instruct the model to answer strictly from that context.</p>

    <table>
      <row header="1"><cell>Approach</cell><cell>Verdict</cell></row>
      <row><cell>Fine-tuning an LLM on our content</cell><cell>Rejected — expensive to retrain every time the site changes, and doesn't reliably stop hallucination.</cell></row>
      <row><cell>Plain prompt with entire site pasted in</cell><cell>Rejected — website content will exceed context limits as pages grow, and is wasteful/costly per request.</cell></row>
      <row><cell><b>RAG (retrieve → ground → generate)</b></cell><cell><b>Selected</b> — small footprint, always answers from current content, easy to update without retraining.</cell></row>
    </table>

    <h2>2.2 How the knowledge base is structured</h2>
    <p>The knowledge base is a set of <b>chunks</b> — short passages (roughly 150–300 words each) cut from our website pages, each stored with the text itself, a vector embedding of that text, and metadata describing where it came from.</p>

    <h2>2.3 How the knowledge base is built (hybrid pipeline)</h2>
    <steps>
      <step title="One-time scrape">A crawler script walks thetatechnolabs.in (services, about, case studies, contact, blog) and pulls clean text per page, stripping nav/footer boilerplate.</step>
      <step title="Chunking">Each page is split into overlapping ~200-word chunks along heading boundaries, so no single chunk loses context mid-sentence.</step>
      <step title="Manual curation">Team reviews the scraped chunks in the admin panel — fixes wording, removes irrelevant boilerplate, adds anything missing from the live site (e.g. FAQs that aren't on the page).</step>
      <step title="Embedding">Each approved chunk is passed through an embedding model and the resulting vector is stored alongside it in Postgres.</step>
      <step title="Freeze as &quot;static&quot;">The knowledge base is now the fixed source of truth. It is only touched again through a deliberate re-scrape or manual edit — never automatically rewritten by user conversations.</step>
    </steps>
