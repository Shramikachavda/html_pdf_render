2. Overall Approach

2.1 Solution Overview

The Theta Chat Bot is designed as an AI-powered Retrieval-Augmented Generation (RAG) system that delivers accurate, context-aware responses using information exclusively retrieved from the Theta Technolabs website.

Rather than storing predefined question-answer pairs or training a custom language model, the chatbot dynamically retrieves the most relevant website content and uses it as contextual input for the Large Language Model (LLM). This enables the chatbot to answer user queries naturally while remaining grounded in verified company information.

This approach also reduces hallucinations by grounding responses in retrieved website content, improves maintainability, and provides flexibility to switch between different LLM providers without redesigning the system.

3.7 Knowledge Synchronization

The chatbot's knowledge base must remain synchronized with the Theta Technolabs website while avoiding unnecessary processing of unchanged content.

Since website content is updated only when new pages, blogs, services, or portfolio items are published, running scheduled synchronization jobs would repeatedly process unchanged data and increase infrastructure overhead.

To address this, the proposed solution adopts an Event-Driven Incremental Synchronization approach. Whenever website content is published, updated, or removed, the synchronization pipeline is triggered automatically. Instead of rebuilding the entire knowledge base, the system identifies only the affected pages and updates those records.

<b>Synchronization Approaches Considered</b>

StrategyDescriptionAdvantagesLimitations
Manual RefreshAdministrator manually re-indexes the entire website.Simple implementationTime-consuming and dependent on manual intervention.
Scheduled SynchronizationAutomatically crawls the website at predefined intervals.Fully automatedProcesses unchanged content and increases unnecessary compute costs.
Event-Driven Incremental Synchronization (Recommended)Triggered whenever website content is created, modified, or deleted. Only affected pages are reprocessed.Efficient, scalable, faster updates, and reduced infrastructure cost.Requires integration with the website publishing workflow or CMS.

<b>Synchronization Workflow</b>

Whenever website content changes, the following pipeline is executed:

1. Detect the newly created or modified webpage.

2. Retrieve the latest page content.

3. Parse and clean the HTML.

4. Remove navigation menus, headers, footers, scripts, banners, and duplicate elements.

5. Normalize the extracted text.

6. Divide the content into semantic chunks.

7. Generate embeddings only for the updated chunks.

8. Replace the existing vectors in the vector database.

9. Update metadata such as indexing timestamp, page version, and source URL.

All other website pages remain untouched.

This approach significantly reduces processing time, minimizes embedding generation costs, and ensures the chatbot always retrieves the most recent website information without reprocessing the complete knowledge base.

The retrieved contextual information is combined with the user's original query to construct a complete prompt, which is then sent to the configured Large Language Model (Google Gemini or OpenAI). The LLM generates a context-aware response based on both the retrieved knowledge and the user's question. Finally, the generated response is returned through the FastAPI backend to the JavaScript chat widget, where it is presented to the user.

This modular workflow separates user interaction, business logic, knowledge retrieval, and AI inference into independent components. Such separation improves maintainability, simplifies testing, enables future technology replacement, and supports incremental system enhancements without requiring significant architectural changes.

3.9 Embeddings: Local Semantic Representation

The chatbot uses Sentence Transformers to generate vector embeddings for both the knowledge base documents and incoming user queries. The embedding model can be selected and optimized based on factors such as retrieval accuracy, model size, hardware availability, and response latency requirements. This flexibility allows the retrieval system to scale while maintaining control over the underlying embedding process.

User input is never treated as an instruction, regardless of phrasing. If a message attempts to override system behavior (e.g., "forget your rules," "act as a different assistant"), the model is instructed to ignore the embedded instruction and respond only using retrieved context for the original question. Regression testing should be performed after:

Prompt modifications.

Website content updates.

Incremental knowledge synchronization.

Embedding model changes.

LLM provider changes.

Backend code changes.

Chunking strategy updates.

A predefined validation dataset should be used to compare system performance before and after each change.

D.9 Test Dataset

A representative set of questions should be maintained as a reference dataset.

The dataset should include questions from all major categories of the Theta Technolabs website, including:

Company information.

Services.

Technologies.

Portfolio.

Industries.

Careers.

Contact information.

General business inquiries.

The same dataset should be reused during regression testing to ensure consistent evaluation.

D.10 Performance Testing

The application should also be evaluated from a performance perspective.

Recommended measurements include:

Average response time.

Retrieval latency.

LLM generation time.

End-to-end processing time.

Concurrent request handling.

API throughput.

These measurements help verify that the chatbot meets expected performance objectives.

D.11 Error & Fallback Testing

The chatbot should be tested using unsupported or ambiguous queries to verify fallback behavior.

Example scenarios:

Questions unrelated to Theta Technolabs.

Incomplete user queries.

Invalid input.

Empty messages.

Questions with no matching website content.

Expected behavior:

Do not generate fabricated information.

Inform the user that the requested information is unavailable.

Recommend visiting the relevant website page or contacting Theta Technolabs through the Contact or Inquiry page.

D.12 Acceptance Criteria

The chatbot will be considered ready for deployment when:

Website content is successfully indexed.

Retrieval consistently returns relevant content.

Chunk quality is verified.

AI responses accurately reflect website information.

Prompt behavior is validated.

Fallback responses operate correctly.

Performance meets project expectations.

No critical functional defects remain.

D.13 Continuous Validation

Testing should not conclude after deployment.

Whenever significant changes occur, the validation process should be repeated.

Examples include:

New website content published.

Changes to the chunking strategy.

Prompt updates.

Migration to a different LLM provider.

Major backend changes.

Knowledge synchronization enhancements.

Continuous validation ensures that chatbot quality remains consistent throughout the application's lifecycle.
