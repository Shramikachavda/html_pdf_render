## 3.1 Introduction

The Theta Chat Bot is a modular AI-powered customer support solution that integrates with the existing Theta Technolabs website without requiring changes to the current Webflow implementation. It follows a service-oriented architecture where the frontend communicates with a FastAPI backend through REST APIs. The backend manages request validation, knowledge retrieval, prompt construction, LLM interaction, and response generation.

The technology stack was selected based on the following criteria:

- **Cost** – Minimize development, deployment, and operational expenses.
- **Control** – Maintain ownership of application data and infrastructure.
- **Development Velocity** – Enable rapid development, testing, and deployment.
- **Scalability** – Support future enhancements such as multilingual capabilities, analytics, and increased user traffic without requiring major architectural changes.

The system is organized into four independent layers — Frontend, Backend, AI & Knowledge Retrieval, and Infrastructure — allowing each layer to be maintained, scaled, or replaced independently.

The architecture is designed to:

- Reuse the existing Webflow website.
- Keep infrastructure lightweight and cost-effective.
- Support future AI provider changes.
- Enable modular feature expansion.
- Simplify deployment using Docker.

## 3.2 Technology Stack Overview

The Theta Chat Bot is built on a modern, modular technology stack that emphasizes scalability, maintainability, performance, and cost-effectiveness. Each technology was selected for its suitability for AI application development, ease of integration, community support, and flexibility for future enhancements. This modular architecture allows individual components to be upgraded or replaced with minimal impact on the overall system.

The complete technology stack is summarized in Table 3.1.

**Table 3.1: Technology Stack Overview**

| Layer | Technology | Role | Reason / Consideration |
|---|---|---|---|
| Chat Interface | Custom JavaScript Widget | Provides the chatbot's user interface and communicates with the backend | Lightweight, framework-agnostic, and easily integrated into the existing Webflow website |
| Programming Language | Python | Core backend and AI application development | Rich AI ecosystem, extensive libraries, and strong community support |
| Backend | FastAPI | Handles REST APIs, request validation, AI orchestration, and business logic | High-performance asynchronous framework with native Python integration, well suited for scalable AI applications |
| AI Model | Google Gemini / OpenAI | Generates natural language responses | Modular design allows switching between LLM providers with minimal code changes |

### Technology Selection Notes

Several components in the proposed architecture offer more than one viable technology option. The final selection for each will be made based on an evaluation of project requirements, performance needs, and operational constraints at the time of implementation. The criteria guiding these decisions are outlined below.

**AI Model Selection:** Two LLM options are considered — Google Gemini and OpenAI. Google Gemini is selected as the initial model for development and evaluation. The final decision between the two will depend on:

- Response quality and accuracy for domain-specific queries
- Latency and performance under expected load
- Operational/API cost at scale
- Alignment with future business requirements

If Gemini does not satisfactorily meet these criteria during evaluation, the modular architecture allows migration to OpenAI with minimal code changes.

**Vector Storage Choice:** Two storage options are considered — Qdrant and PostgreSQL with pgvector. The selection between them will depend on:

- **Qdrant** — preferred if the priority is high-performance, large-scale vector search as a standalone requirement. If custom dashboards are not required, Qdrant can be used on its own without the added overhead of managing a separate relational database.
- **PostgreSQL + pgvector** — preferred if structured application data (chat history, user sessions, analytics, dashboards) needs to be managed alongside vector embeddings within a single database.

The final choice will be made based on which factor — pure search performance vs. unified data management — is more critical to the deployment.

**Deployment Flexibility:** The system is designed for on-premise deployment by default, prioritizing cost control. If on-premise resources prove insufficient, or scalability needs grow, the architecture supports migration to cloud platforms.

## 3.3 Technology Selection Rationale

Instead of selecting technologies based solely on popularity, each major component was evaluated against multiple alternatives using criteria such as performance, scalability, maintainability, operational cost, ease of deployment, and compatibility with AI-based applications. The following sections summarize the key architectural decisions made during the technology selection process.

### 3.3.1 Backend Framework Selection

| Technology | Advantages | Limitations |
|---|---|---|
| Flask | Lightweight and simple | Limited support for asynchronous APIs |
| Django | Feature-rich framework | Overhead for AI microservices |
| Node.js (Express) | Good performance | Smaller AI ecosystem than Python |
| **FastAPI (Selected)** | High performance, asynchronous processing, automatic API documentation, strong AI ecosystem | Requires understanding of asynchronous programming |

**Decision:** FastAPI was selected because it provides high-performance asynchronous request handling, integrates seamlessly with Python AI libraries, and automatically generates OpenAPI documentation. These capabilities make it well suited for scalable AI-powered REST services.

### 3.3.2 Large Language Model Selection

| Provider | Advantages | Limitations |
|---|---|---|
| Google Gemini | Cost-effective, large context window, good performance | May require evaluation for highly complex reasoning |
| OpenAI | Excellent reasoning and structured output | Higher operational cost |

**Decision:** Google Gemini is proposed as the initial LLM for development and evaluation due to its balance between performance and operational cost. However, the application architecture remains provider-independent, allowing migration to OpenAI if future evaluation indicates improved performance or business value.

### 3.3.3 Vector Storage Selection

| Technology | Advantages | Limitations |
|---|---|---|
| Qdrant | High-performance semantic search, lightweight, Docker-friendly | Separate database for structured data |
| PostgreSQL + pgvector | Unified relational and vector storage, supports analytics | Higher infrastructure requirements |

**Decision:** The final selection depends on deployment requirements. Qdrant is recommended for deployments focused on semantic search, while PostgreSQL with pgvector is better suited for deployments requiring dashboards, reporting, analytics, and unified data management.

### 3.3.4 Deployment Strategy

| Option | Advantages | Limitations |
|---|---|---|
| On-Premise | Data privacy, infrastructure control, lower long-term cost | Requires internal server management |
| Cloud | Elastic scalability and managed services | Ongoing infrastructure costs |

**Decision:** On-premise deployment is recommended as the default deployment model to maximize infrastructure control and data privacy. The modular architecture also supports migration to cloud infrastructure if future scaling requirements justify the transition.

## 3.4 Component Interaction

The Theta Chat Bot follows a modular request-processing workflow in which each component is responsible for a specific stage of the chatbot lifecycle. User interactions begin through a custom JavaScript chat widget embedded within the existing Theta Technolabs website, which is developed using Webflow. The widget communicates with the backend through secure REST APIs exposed by the FastAPI application.

Upon receiving a user request, the FastAPI backend validates the request and performs the necessary preprocessing. The user's query is then converted into a vector embedding using the selected embedding model (Sentence Transformers). This embedding is used to perform semantic similarity search against the vector database (Qdrant or PostgreSQL with pgvector) to retrieve the most relevant information from the chatbot's knowledge base.

The retrieved contextual information is combined with the user's original query to construct a complete prompt, which is then sent to the configured Large Language Model (Google Gemini or OpenAI). The LLM generates a context-aware response based on both the retrieved knowledge and the user's question. Finally, the generated response is returned through the FastAPI backend to the JavaScript chat widget, where it is presented to the user.

This modular workflow separates user interaction, business logic, knowledge retrieval, and AI inference into independent components. Such separation improves maintainability, simplifies testing, enables future technology replacement, and supports incremental system enhancements without requiring significant architectural changes.

## 3.5 APIs Used

| API | Type | Purpose |
|---|---|---|
| **Internal — `/api/chat`** | REST (FastAPI, own API) | The only endpoint the frontend widget calls; accepts a user message, returns the generated answer + sources |
| **Gemini API** | External | Primary LLM — generates the natural-language response from the assembled prompt |
| **OpenAI API** *(optional)* | External | Fallback LLM, used case-by-case when a Gemini response fails the quality bar during testing |
| **Gemini Embedding API** | External | Default embedding generator — converts knowledge-base chunks and user queries into vectors |
| **Qdrant REST API** | Internal (self-hosted) | Stores vectors, performs similarity search; called by the backend, never by the frontend |

> Sentence Transformers, if used instead of the Gemini Embedding API, runs **locally** inside the backend — it is not an external API call, which is precisely its advantage (no per-token cost, works offline).

## 3.6 Frontend: Integrating with Webflow Using an Embedded Widget

The Theta Technolabs website is already built and maintained using Webflow. To preserve the existing website while adding AI chatbot functionality, the chatbot is implemented as a lightweight embedded JavaScript widget. This approach allows the chatbot to integrate seamlessly with the current website without requiring modifications to the existing Webflow structure or design.

Benefits of this approach:

- Integrates with the existing Webflow website without structural changes.
- Simple deployment using a single JavaScript `<script>` embed.
- Maintains the existing website design and content management workflow.
- Allows independent customization of the chatbot's appearance, including colors, position, and branding.
- Enables future updates to the chatbot without affecting the website implementation.

## 3.7 Backend: Python + FastAPI

**Python** was the obvious base layer — it's where the AI ecosystem lives (embedding SDKs, vector-DB clients, LLM SDKs), and it keeps one language across the whole backend instead of splitting logic between a JS API layer and a Python ML layer.

**FastAPI** sits on top of it because the bot's job is fundamentally I/O-bound: call an embedding model, query the vector store, call an LLM, return a response. FastAPI's async request handling means one worker can juggle many concurrent chats without blocking, and its automatic OpenAPI docs mean the API is self-documenting for whoever maintains it next.

| Alternative | Why it lost |
|---|---|
| Node.js (Express) | Solid for APIs, but a thinner AI/ML library ecosystem |
| Flask | Simple, but synchronous by default — a poor fit for concurrent LLM calls |
| Java / .NET | Reliable for enterprise systems, but slower to iterate on AI-integration code |

## 3.8 AI Model Layer: Gemini First, Provider-Agnostic by Design

The response-generation layer is built behind a thin provider interface, so the LLM is a **configuration value, not a hardcoded dependency**. If Gemini underperforms on a specific use case later, swapping to OpenAI doesn't touch the retrieval pipeline, the API contract, or the frontend.

| Criterion | Gemini | OpenAI |
|---|---|---|
| Cost per token | Lower | Higher |
| Context window | Larger | Large |
| Reasoning depth | Strong | Excellent |
| Structured output reliability | Good | Excellent |
| Fit for this project (retrieval-grounded Q&A) | Best default | Strong fallback |

**Testing approach:** Gemini is used first for every new feature or prompt. If a response during testing doesn't meet the quality bar (accuracy, tone, or completeness), the same test case is re-run against OpenAI as a comparison pass before deciding whether to switch providers for that use case. This keeps the evaluation evidence-based rather than a blanket provider swap.

## 3.9 Embeddings: Local Semantic Representation

The chatbot uses Sentence Transformers to generate vector embeddings for both the knowledge base documents and incoming user queries. The same embedding model is used during document indexing and query processing to ensure that both are represented within the same vector space, enabling accurate semantic similarity search.

Sentence Transformers provides a locally deployable embedding solution that eliminates dependency on external embedding APIs and avoids recurring API costs. Since embeddings are generated within the organization's infrastructure, this approach provides better control over data privacy and supports offline processing.

The embedding model can be selected and optimized based on factors such as retrieval accuracy, model size, hardware availability, and response latency requirements. This flexibility allows the retrieval system to scale while maintaining control over the underlying embedding process.

## 3.10 Vector Database: Qdrant, or PostgreSQL + pgvector

| Option | Strength | Weakness |
|---|---|---|
| **Qdrant (default)** | Open-source, metadata filtering, Docker-native, production-ready, no relational-DB overhead | Self-managed; separate database if structured data is also needed |
| **PostgreSQL + pgvector** | Vectors and structured data (chat history, sessions, analytics) live in one database | Higher infrastructure requirement than Qdrant alone |

**Decision rule:** Qdrant if the priority is pure semantic search with nothing else to manage. PostgreSQL + pgvector if the project needs to store chat history, sessions, or analytics alongside vectors — see the hosting note below for the full trade-off.

## 3.11 Containerization: Docker + Docker Compose

Every service — the FastAPI backend, the vector store, and Nginx — ships as a container, defined once in a `docker-compose.yml`, so "works on my machine" stops being a risk. Identical environments across dev, staging, and production; one-command startup (`docker compose up`); simple rollback by pinning image tags.

## 3.12 Hosting: On-Premise Linux, With a Documented Exit Ramp

The recommendation is an on-premise Linux server rather than a cloud platform — driven by data control, not cost alone. Company and customer conversation data never leaves internal infrastructure, and there's no recurring cloud compute bill for a workload that's steady rather than spiky.

> **Note — hosting is not fixed:** if the on-premise server doesn't have enough disk/compute headroom, the hosting model can shift without a redesign. The right path depends on one question: **do we need to store what users ask?**
>
> **Option A — Don't store user queries.** Keep Qdrant as the vector store and add Prometheus + Grafana for monitoring (uptime, latency, request volume). This needs an extra custom container for the monitoring stack and some initial setup work, and still consumes on-premise space for those containers — but no conversation data is retained.
>
> **Option B — Store user queries** (for analytics, QA, or improving answers over time). Move to **PostgreSQL + pgvector** with Python as the access layer, replacing Qdrant. This sets up a natural next step: a **custom admin panel** built later on top of the same Postgres instance for reviewing conversations and reporting.
>
> Both options stay on-premise; if space is the real constraint rather than the storage model, either can also move to a cloud-hosted instance without changing anything else in the architecture.
